1. Overview

The IBM Onboarding Platform is an internal product designed to replace ad-hoc onboarding spreadsheets, static intranet pages, and disconnected email chains with a single, living platform that adapts to each organisation, team, and individual.

Think of it as Jira for onboarding — every task is tracked, every user has a personalised board, managers get real-time visibility, and HR admins can configure the entire experience without writing code.

For New Starters

A personalised task board showing exactly what to do, in what order, and by when — with AI help on demand.

For Managers

Real-time view of every direct report's onboarding progress. Assign tasks, personalise goals, flag blockers.

For HR & Admins

Full control over what modules are active, what tasks exist, what documents are linked, and what the platform looks like.

For Organisations

Plugin-based architecture means the platform fits any org structure — financial services, consulting, product, or public sector.

New starter onboarding at IBM (and most large organisations) suffers from three consistent failure modes:

3. Architecture

The platform is built in five layers, each independently deployable and testable.

UI Layer

Next.js 14 (App Router). New starter dashboard, manager view, admin config portal, analytics. Server-side rendered for performance.

API Layer

Node.js + Express REST API. Handles user state, task CRUD, org config, plugin registry, auth middleware.

Plugin Engine

A registry of self-contained modules (checklist, training, FAQ, etc.) that orgs enable and configure independently. See §4.

Data Layer

PostgreSQL for persistent state. Redis for session cache and notification queues. All schemas versioned with migrations.

4. Plugin System

Every section of the dashboard is a plugin — a self-contained module with its own config schema, UI component, and optional API integration. Org admins enable, disable, reorder, and configure plugins via the Admin Portal with no code changes required.

Plugin contract: Every plugin exports a PluginDefinition object with an id, display name, config schema (JSON Schema / Zod), a default config, and a React component that receives resolved config as props.

Plugin Config Schema Example

Each org's configuration is a single JSON document stored in the database. Here's an example for the checklist plugin:

{
  "pluginId": "checklist",
  "enabled": true,
  "order": 1,
  "config": {
    "title": "First Week Checklist",
    "tasks": [
      {
        "id": "setup-laptop",
        "label": "Receive and set up IBM laptop",
        "category": "IT",
        "dueDayOffset": 1,
        "mandatory": true,
        "roles": ["all"],
        "link": { "label": "Setup guide →", "url": "/docs/laptop-setup" }
      }
    ]
  }
}

6. Task State Model

Every task assigned to a user has a persistent state record. This is the core data model — everything else is built on top of it.

Task Lifecycle

NOT_STARTED  →  IN_PROGRESS  →  BLOCKED  →  DONE  →  VERIFIED
                     ↑                          ↓
                     └──────────── REOPENED ────┘

Key Fields (per user-task record)

FieldTypeDescription
user_idUUIDThe new starter this record belongs to
task_idstringPlugin-scoped task identifier (e.g. checklist:setup-laptop)
statusenumCurrent state in the lifecycle above
due_datedateCalculated from start_date + due_day_offset
completed_attimestampWhen the task was marked Done
verified_byUUIDManager/system that verified completion (nullable)
evidence_urlstringLink to uploaded certificate or external proof (nullable)
notestextFree-text notes from the user or their manager
Overdue detection: A background job runs nightly to flag tasks where due_date < today and status != DONE. Affected managers and HR admins receive a digest notification.

9. Getting Started

Prerequisites: Node.js 22+, Docker Desktop, access to IBM GitHub Enterprise, and an IBM w3id account for authentication.

Clone & Install

# Clone the repo
git clone https://github.ibm.com/future-now/onboarding-platform.git
cd onboarding-platform

# Install all workspace dependencies
npm install

Environment Setup

# Copy the example env file
cp .env.example .env.local

# Required variables
DATABASE_URL=postgresql://localhost:5432/onboarding
REDIS_URL=redis://localhost:6379
OIDC_CLIENT_ID=your-w3id-client-id
OIDC_CLIENT_SECRET=your-w3id-client-secret   # store in vault, not .env
WATSONX_API_KEY=your-watsonx-key             # store in vault, not .env

Run Locally (Dev)

# Start PostgreSQL + Redis via Docker
docker compose up -d db redis

# Run DB migrations
npm run db:migrate

# Start both API and web in parallel
npm run dev

Run Tests

npm run test          # unit + integration (Vitest)
npm run test:e2e      # end-to-end (Playwright)
npm run lint          # ESLint + Prettier check
npm run typecheck     # TypeScript strict check

10. Contributing

Branch Strategy

BranchPurpose
mainProduction-ready code. Protected — PRs only.
developIntegration branch. All feature PRs target here.
feature/*New features (e.g. feature/slack-plugin)
fix/*Bug fixes
chore/*Non-code changes (docs, config, deps)

PR Checklist

  • Tests written and passing (npm run test)
  • TypeScript strict check passes (npm run typecheck)
  • Lint clean (npm run lint)
  • No secrets or credentials committed
  • If adding a plugin: PluginDefinition exported, config schema defined, README entry added
  • If changing DB schema: migration file included
  • PR description explains what, why, and how to test

Adding a New Plugin

All plugins live in packages/plugins/src/. Create a new folder and export a PluginDefinition:

// packages/plugins/src/my-plugin/index.ts
import { z } from 'zod';
import { PluginDefinition } from '../types';

export const myPlugin: PluginDefinition = {
  id: 'my-plugin',
  name: 'My Plugin',
  category: 'custom',
  configSchema: z.object({
    title: z.string().default('My Section'),
    items: z.array(z.string()).default([]),
  }),
  component: MyPluginComponent,   // React component
  defaultConfig: { title: 'My Section', items: [] },
};

11. Security & Compliance

This platform handles employee PII. All contributors must follow IBM security standards. No exceptions.
RequirementImplementation
AuthenticationOpenID Connect via IBM w3id. No local credential storage.
AuthorisationRole-based access control (RBAC) enforced on every API route. No client-side-only checks.
Data in transitTLS 1.3 required for all connections. No HTTP in production.
Data at restAES-256 encryption for sensitive fields. Database encryption at rest on IBM Cloud.
SecretsAll credentials via environment variables or IBM Key Protect. Never committed to Git.
ContainersRed Hat UBI9 Minimal base image. Non-root user. Read-only filesystem where possible.
DependenciesMend scanning on every PR. No packages with known High/Critical CVEs.
LoggingStructured JSON logs. No PII, passwords, or tokens in log output.
GDPRData retention policies enforced. Right-to-erasure endpoint provided. DPA completed.
